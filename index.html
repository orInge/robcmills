Concept

So most traditional tile-based landscape paging methods will deallocate and delete geometry as the player moves away from it, and allocate 'new' memory for the geometry that is loaded on the edge the player is moving towards. I thought, "Why not just change the values of existing vertices?" Changing the positions is rather straight-forward, as well as the texture coordinates. But the challenge lay in the reordering of the indices. For every strip of vertices you 'shift' from one edge to the opposite, all those triangles are broken. How hard would it be to figure out how to reorder the indices?

Benchmarking faster than the Monk!

Another fun problem from RubyMonk.com solved. This one is called Number Shuffle and much to my surprise the Monk's provided solution seemed fairly inefficient at first glance. First I noticed the magic constants 3 ? 6 : 24 Ugly, and degeneralizing, but ok whatever. But why would he use Array:shuffle and :uniq to find what is apparently a permutation? Isn't that like using a slot machine repeatedly until you get 24 jackpots? Anyway, the proof is in the performance numbers:

Problem:

Give a 3 or 4 digit number with distinct digits, returna sorted array of all the unique numbers that can be formed with those digits.

Example:

Given: 123 Return: [123, 132, 213, 231, 312, 321] 


require 'benchmark'

def my_number_shuffle(num)
  digits = num.to_s.split(//)
  digits.map!(&:to_i)
  perm_a = digits.permutation.to_a
  ret = perm_a.collect{ |a| a.join.to_i }
  ret.sort!
end

def monk_number_shuffle(num)
  no_of_combinations = num.to_s.size == 3 ? 6 : 24
  digits = num.to_s.split(//)
  combinations = []
  combinations << digits.shuffle.join.to_i while combinations.uniq.size != no_of_combinations
  combinations.uniq.sort
end

my_time = Benchmark.realtime do
  1000.times { my_number_shuffle(123) } 
end

monk_time = Benchmark.realtime do
  1000.times { monk_number_shuffle }
end

puts "My time: #{my_time} seconds"
puts "Monk time: #{monk_time} seconds"
